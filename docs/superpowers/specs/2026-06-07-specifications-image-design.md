# 规格尺寸图升级设计

> 已被 `2026-06-07-required-specifications-input-design.md` 替代。当前实现要求用户在调用时直接提供尺寸，并严格交付完整七图。

## 目标

在现有六类电商产品图基础上，新增默认第七类“规格尺寸图”。该图片使用用户提供的真实规格数据，以经典电商参数布局展示商品、尺寸标线、参数表和测量免责声明。

默认交付仍为 `1:1` 方图。除非用户明确要求其他比例，否则不额外生成竖图或横图。

## 默认交付

完整套系由六张改为七张：

1. `01-main-square.png`
2. `02-atmosphere-square.png`
3. `03-detail-square.png`
4. `04-meaning-copy-square.png`
5. `05-usage-square.png`
6. `06-selling-points-square.png`
7. `07-specifications-square.png`

第七张的 manifest 类型为 `specifications`。

## 参数来源

规格尺寸图只使用以下信息：

- 用户通过文字明确提供的数据。
- 用户上传的现有参数图中能够清晰读取、且经过用户确认的数据。

产品照片可用于识别尺寸标线应指向的结构位置，但不得用于推算尺寸、容量、重量、材质纯度、品牌、型号或工艺。

接受灵活的 `字段：内容` 输入，不强制固定字段集合。常见字段包括：

- 尺寸：长、宽、高、口径、底径、直径、厚度。
- 参数：品牌、材质、容量、重量、工艺、型号。

至少需要一项尺寸字段才能生成规格尺寸图。数值、单位及“约”等限定词按用户原文使用，不自动换算、补全或修正。

## 缺失与冲突处理

用户未提供尺寸参数时，Skill 主动询问一次，并给出结构化填写示例。

如果用户仍未补充：

- 前六张继续正常生成和交付。
- 第七张不生成猜测版本。
- manifest 中第七张保持 `status: "pending"`。
- 第七张增加 `availability: "awaiting-user-input"`。
- `notes` 写入 `等待用户提供规格参数`。
- 质量报告和最终回复明确说明第七张待补充。

校验器只在 `type: "specifications"`、`status: "pending"`、`availability: "awaiting-user-input"` 且备注完整时允许第七张缺失。其他图片的 `pending` 状态仍视为错误，避免条件性交付掩盖真正未完成的资产。

用户提供的信息互相冲突，或多张图疑似属于不同 SKU 时，暂停并请求确认，不自行选择数据。

## 视觉设计

采用已确认的 A 版经典参数表布局：

- 顶部：两列或自适应列数的参数表。
- 中部：完整、清晰、严格保真的商品主体。
- 商品周围：与对应结构对齐的尺寸线、端点和标签。
- 底部：测量免责声明。

默认免责声明为：

`尺寸为手工测量，存在合理误差，请以实物为准`

参数较少时保持充足留白，不为填满版面虚构字段。参数较多时优先保证文字可读和商品完整，不把文字压在商品关键细节上。

## 生成方式

按用户确认，规格图采用纯 ImageGen 一体生成：

- 商品、参数表、尺寸线、数字、单位和免责声明由一次专用 ImageGen 调用完成。
- 使用原始商品图作为身份参考。
- 提示词逐字列出所有参数，不允许模型增删字段、改写数字或替换单位。
- 每个尺寸标签必须通过标线连接到正确结构。

不新增 OCR、Canvas 或后期文字排版脚本。

## 质量检查与重试

规格尺寸图除现有商品保真检查外，还必须检查：

- 每个字段名称和值逐字正确。
- 数字、小数点、单位和“约”字逐字正确。
- 尺寸线指向正确部位，且不存在互相矛盾的标注。
- 参数表没有重复、漏项或模型自行增加的字段。
- 商品没有被尺寸线或文字遮挡。
- 免责声明完整、清晰。

最多进行两次定向重试，每次只修复明确失败项，同时重复全部真实参数和商品不变量。

两次后仍有错字、错数值、错单位或错误标线时：

- 不把错误图片标记为完成。
- manifest 中第七张标记 `status: "failed"`。
- 在 `quality-report.md` 和最终回复中列出失败原因。

## 脚本与 Manifest

`scripts/create_delivery.py` 将默认创建七条资产记录。新增命令参数：

```text
--specifications-awaiting-input
```

Skill 只有在询问一次后用户仍未提供参数时才传入该参数。传入时，第七条资产记录写入：

```json
{
  "id": "07-specifications",
  "type": "specifications",
  "status": "pending",
  "availability": "awaiting-user-input",
  "notes": ["等待用户提供规格参数"]
}
```

未传入时，第七张与其他资产一样以普通 `pending` 状态创建，并必须在最终验证前完成或明确失败。

manifest schema 版本从 `1` 升至 `2`。

## 验证规则

`scripts/validate_delivery.py` 支持两种成功交付：

1. 七张图片全部为 `complete`，所有文件和比例有效。
2. 前六张全部为 `complete`，第七张满足严格的 `awaiting-user-input` 条件。

以下情况必须验证失败：

- 第七张普通 `pending`，但没有条件性缺失元数据。
- 第七张标记等待输入，却存在虚构或占位图片。
- 用户已有参数时第七张文件缺失。
- 任意前六张缺失、失败或未完成。
- 图片比例不符合 manifest。

验证成功信息应区分“7 张完整交付”和“6 张完成、1 张等待参数”。
